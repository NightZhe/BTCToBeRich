# 40 - 維護協議（未來的模型如何安全地更新這套制度）

原則：制度要活，但不能亂長。改之前先分清你要做的動作屬於哪一級。

---

## 1. 權限分級

**A 級——可自行改，不用問使用者：**
- 在 `lessons.md` 末尾追加教訓（格式見第 3 節）。
- 更新 CLAUDE.md 專案清單（新專案出現、路徑改變、上線網址變更）。
- 修正**查證過確實錯誤**的路徑、指令、工具名、模型參數
  （例如 session 開頭系統注入的清單與 `10-dispatch.md` 第 2 節不符 → 以注入為準更新，
  並在該節更新查證日期）。
- 錯字、格式修正。

**B 級——先向使用者說明再改：**
- 刪除或放寬任何規則（門檻數字、硬規則、驗收標準）。
- 改變檔案結構（合併/拆分/重新命名規則檔）。
- 動 `profile.md` 的個人資料內容。
- 執行第 4 節的整併。

**C 級——不可做：**
- 刪除 `.claude/backups/` 裡的備份。
- 把一次性任務指令寫成全域規則（這正是舊 CLAUDE.md 的病，見 00-diagnosis.md 問題 2）。
- 未經使用者要求，把個人/財務資料寫進任何規則檔或會 commit 的檔案。

## 2. 改檔流程（A、B 級都適用）
1. 備份：`cp <檔案> /Users/ben/harryaiagent/.claude/backups/<檔名>.$(date +%Y%m%d-%H%M).bak`
2. 修改。
3. Read-back 自查：重讀改動段落，確認沒有破壞表格/清單結構、沒有與同檔其他規則矛盾。
4. B 級加一步：在給使用者的回覆中說明「改了什麼、為什麼」。

## 3. 教訓回寫（每次踩坑後，5 分鐘內完成，A 級）

寫進 `lessons.md` 末尾，固定格式：

```
## YYYY-MM-DD | 一句話標題
- 情境：當時在做什麼
- 錯誤：具體發生了什麼（貼關鍵錯誤訊息或行為）
- 教訓：一句話，下次怎麼做
- 建議規則變更：無 ／ 或「10-dispatch.md 第 X 節應加上…」（B 級，先問再改）
```

什麼算「坑」值得寫：同一錯誤第二次發生、使用者糾正了你的做法、驗收發現執行者
宣稱完成但實際沒完成、規則檔指引了錯誤方向。
什麼不用寫：一次性的手滑、已被現有規則涵蓋的錯（那代表你沒讀規則，重讀就好）。

## 4. 整併觸發（防止 lessons 變垃圾場）
- 平時**不需要**讀 `lessons.md`（它是寫入用的）。
- 當 `lessons.md` 超過 20 條或 150 行 → 觸發整併（B 級，先告知使用者）：
  1. 派一個 opus agent 讀全部 lessons，找出重複出現的模式。
  2. 重複 ≥2 次的教訓 → 升級成規則，寫進對應規則檔（附「源自 lessons YYYY-MM-DD」）。
  3. 已升級與過時的條目移到 `lessons-archive.md`，`lessons.md` 只留未整併的。

## 5. 體積上限（防止制度反過來吃 context）
- CLAUDE.md ≤ 80 行。超了就把內容抽到 rules/ 檔案，CLAUDE.md 留一行路由。
- 每個規則檔 ≤ 200 行。超了先刪冗餘，再考慮拆檔（拆檔是 B 級）。
- 規則檔之間不互相複製內容，只引用檔名＋節號。

## 6. 記憶機制分工（避免三處重複記錄）
- **使用者的長期事實／偏好回饋**（他是誰、他喜歡怎樣的回覆）→ 系統提示描述的
  記憶目錄（memory/，含 MEMORY.md 索引）。2026-07-07 查證時該目錄是空的，可以開始用。
- **工作制度／流程規則**（怎麼派工、怎麼驗收）→ `.claude/rules/`。
- **CLAUDE.md** → 只放路由與硬規則。
- 判斷方法：內容是「關於使用者」→ 記憶目錄；「關於怎麼工作」→ rules/。
