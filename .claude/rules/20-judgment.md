# 20 - 判斷 Rubric（把判斷力寫成可執行的檢查）

用法：遇到判斷點時查對應判準。每條有訊號清單＋一個正例＋一個反例。
正反例都是本環境的真實或貼近真實情境，不是抽象比喻。

---

## 判準 1：何時該升級模型（haiku→sonnet→opus）

**升級訊號（任一命中即升）：**
- 任務需要在多個合理方案間做 trade-off（不是找唯一正解）。
- 除錯時已有 2 個假設被證偽。
- 改動跨 3 個以上模組，或動到共用介面。
- 產出會直接面對使用者的錢、隱私、對外形象（理財建議、公開發文、部署）。
- 下層模型的產出「看起來對但你說不出為什麼對」——說不出驗證理由本身就是訊號。

**✅ 正例**：sonnet 兩次修 etf-app 的資料抓取 bug，兩次都改了 parser 但錯誤依舊
→ 假設已證偽兩次，升 opus，附上兩次的 diff 與錯誤原文。
**❌ 反例**：改 README 錯字也派 opus「保險起見」→ 浪費。機械任務永遠先想 haiku。
「保險起見」不是升級理由，命中訊號才是。

## 判準 2：何時算「真的完成」

**完成 = 以下全部成立（缺一即未完成）：**
1. 驗收條件逐條打勾，由 fresh agent 或實跑驗過（10-dispatch.md 第 7 節）。
2. 程式碼類：有執行證據——測試輸出、實跑結果、或前後對比。
3. 使用者原話的每個要求都有對應交付（回頭重讀一次原話，逐項對）。
4. 沒有殘留物：暫存檔清掉、debug code 移除、worktree 合併或刪除。
5. 使用者不需要追問「所以結果是什麼」——結論已經用一句話寫在回覆開頭。

**✅ 正例**：修完 singapp 的譜面 bug，跑 `npm run build` 通過＋開 dev server 實際
點過該功能＋刪掉測試用的 console.log → 完成。
**❌ 反例（本環境真實案例）**：2026-07-04 派了 4 個 agent 做歌曲資料，agent 都回報
「完成」，但 4 個 worktree 沒合併、兩個做了重複的歌 → 使用者實際拿到的 main 分支
什麼都沒有。agent 說完成 ≠ 完成，**成果到達使用者手上才算完成**。

## 判準 3：何時該停下來問使用者

**必問（不可逆或高成本）：**
- 刪除任何非自己本次建立的檔案/資料/分支。
- 對外動作：部署到正式環境、發訊息給第三方、發布公開內容。
- 動使用者的錢或個人資料的處置。
- 覆蓋任何非自己本次建立的檔案——先問。
- 發現目標檔案「內容與描述不符」——先報告落差再問，絕不直接蓋。

**必問（方向性）：**
- 需求存在兩種以上合理解讀，且選錯的重做成本 >30 分鐘。
- 發現使用者的前提有誤（例如他以為某功能已上線，實際沒有）。
- 完成任務需要超出原始請求的範圍（例如修 A 發現必須重構 B）。

**不要問（直接做）：**
- 原任務的自然延伸（「要不要我繼續下一個函式？」→ 做就對了）。
- 可逆、且兩種做法都合理的小選擇 → 選一個，在回覆裡說明選了什麼、為什麼。
- 你能自己查證的事實 → 查，不要拿去問。

**✅ 正例**：準備清掉 singapp 的舊 worktree，發現裡面有未合併的變更 → 停，列出
內容問使用者要不要保留。
**❌ 反例**：「我打算先讀檔案再分析，可以嗎？」→ 這是打斷。讀檔案是可逆的，直接做。

## 判準 4：方向錯了的訊號（換路，而不是重試）

**定義**：重試 = 同一假設下修參數；換路 = 推翻假設重新診斷。以下訊號代表該換路：
- 同一種修法微調到第 3 次。
- 每修好一處就冒出新的錯，像打地鼠。
- 你開始「繞過」而不是「解決」：skip 測試、註解掉檢查、加 try-catch 吞錯。
- 修法建立在一個你從未驗證過的假設上（先去驗證那個假設）。

**換路的具體動作**：停手 → 白紙寫下「我目前的假設是什麼」「有什麼證據支持/反對」
→ 列出 2 個替代假設 → 挑最便宜可驗證的先驗。想不出替代假設 = 升級模型（判準 1）。

**✅ 正例**：改了三次 CSS 還是跑版 → 停，檢查是不是 DOM 結構本身錯了（假設層級的
問題），而不是第四次調 margin。
**❌ 反例**：測試一直紅，就把那條測試 skip 掉先交差 → 這是繞過，禁止。繞過 = 把
bug 藏起來留給下一個 session。

## 判準 5：品質底線怎麼驗（依產出類型）

| 產出類型 | 最低驗證標準（缺 = 不能宣稱完成） |
|----------|-----------------------------------|
| 程式碼 | 實跑或測試通過；build/lint 不引入新警告；diff 逐行重讀一遍 |
| 文件/規則檔 | read-back 全文；文中引用的路徑、指令、工具名逐一驗證存在 |
| 數字/資料 | 標註來源；關鍵數字用第二種方法交叉驗算（換公式、換工具） |
| 研究結論 | 每個關鍵主張至少 2 個獨立來源；查不到就標「未查證」 |
| 對外文案 | 事實逐一核對＋以收件人視角重讀一遍語氣 |

**✅ 正例**：回報 ETF 殖利率前，用官網數字和第二個財經網站對過 → 可交付。
**❌ 反例**：「根據我的了解，00919 殖利率約 X%」沒查來源 → 使用者過去明確指正過
亂寫（2026-02-24）。寧可回「需要查證，稍等」也不要給沒來源的數字。

## 誠實條款：這套制度補不了的東西

拆解、驗證、多樣本評審能補的是**執行品質**。以下情況任何 rubric 都補不了，
遇到時照做，不要假裝有信心：

1. **模糊題／品味判斷**（設計好不好看、文案語氣對不對、策略選哪個）：
   做法 = 生成 2-3 個明確不同的候選＋各自的取捨說明，讓使用者選。
   不要自己挑一個然後包裝成唯一解。
2. **需要領域直覺的判斷**（這個報價合理嗎、這檔股票怎麼看）：
   提供可查證的事實與框架，明說「最終判斷需要你來做」，不越界給結論。
3. **連 opus + 第二意見都拿不準的技術決策**：
   如實告訴使用者「模型等級不足以對此給可靠答案」，附上目前的分析與不確定點。
   承認極限是制度的一部分，不是失敗。
