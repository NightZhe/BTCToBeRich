# 30 - 派工 Prompt 模板

用法：派 subagent 時，複製對應模板，把 `【】` 填滿後放進 Agent 工具的 `prompt`。
`subagent_type` 和 `model` 按模板開頭的建議填（升降級規則見 10-dispatch.md 第 6 節）。
所有模板已內建三件套（目標與動機／驗收條件／回報格式），不要刪這三段。

通用規則：
- 【已知路徑】填得越全，agent 浪費在探索的 token 越少。真的不知道就寫「未知，自行探索」。
- 驗收條件寫不出來 = 任務還沒想清楚，先想清楚再派。
- 多 agent 平行派工時，每個 agent 的【範圍】必須互斥，並在各自 prompt 裡寫明
  「以下部分由其他 agent 負責，你不要做：【】」。

---

## 模板 1：搜尋／定位（subagent_type: Explore, model: haiku）

```
目標：找出【要找什麼：功能實作位置／設定來源／某字串的所有使用處】。
動機：我接下來要【下一步要做的事】，需要先知道位置。

範圍：【目錄，例如 ~/harryaiagent/singapp/src】。跳過 node_modules、dist、build 產物。
已知線索:【關鍵字、檔名片段、相關檔案路徑；沒有就寫「無」】

回報格式（總長 30 行以內）：
1. 每個命中：檔案:行號 ＋ 一句話說明這裡是做什麼的
2. 一句話結論：核心邏輯在哪、入口在哪
3. 沒找到的話：列出你搜過的關鍵字與目錄，不要猜一個位置回報
```

## 模板 2：實作（subagent_type: general-purpose, model: sonnet）

```
目標：【要做什麼，一句話】。
動機：【為什麼要做——讓你在細節取捨時知道什麼重要】。

相關檔案：【已知路徑清單】
範圍邊界：只改【允許動的檔案/目錄】。不要動【禁區，例如共用介面、設定檔】。
不要引入新依賴；需要新套件時停下來在回報中說明理由，不要直接裝。
風格：跟隨周圍既有程式碼的命名與慣例，不要重新發明。

驗收條件（完成前逐條自查）：
- [ ]【功能行為，例如「頁面 X 顯示 Y」】
- [ ]【執行證據，例如「npm run build 通過」「測試 X 綠燈」——必填，宣稱『應該可以』不算】
- [ ] 沒有留下 debug code、註解掉的舊碼、暫存檔

回報格式（30 行以內）：
1. 結論：做了什麼、驗證方式與結果
2. 改動清單：每個檔案:行號 ＋ 一句話
3. 未解事項與風險（沒有就寫「無」）
禁止貼大段程式碼回來；diff 留在檔案裡。
```

## 模板 3：重構（subagent_type: general-purpose, model: sonnet；跨模組或動共用介面時用 opus）

```
目標：重構【對象】：【現狀問題 → 目標狀態】。
動機：【為什麼值得重構】。
鐵律：外部行為不可改變。這是重構，不是改功能。

相關檔案：【路徑清單】
範圍邊界：【允許動的範圍】。公開介面/匯出簽名【可以／不可以】改。

驗收條件：
- [ ] 重構前先跑一次現有測試/build 記錄基準，重構後結果一致
- [ ]【沒有測試時：寫一個最小行為驗證腳本，前後各跑一次，輸出一致】
- [ ] 舊的死程式碼已刪除，不是註解掉

回報格式（30 行以內）：
1. 結論：改了什麼結構、行為不變的證據（前後輸出對比）
2. 改動清單：檔案:行號 ＋ 一句話
3. 你發現但沒動的問題（超出範圍的），列出來給主對話決定
```

## 模板 4：研究（subagent_type: general-purpose, model: sonnet）

```
研究問題：【具體問題，含判斷標準，例如「X 和 Y 哪個適合 Z 場景，考量 A、B」】。
動機：【這個答案會用來決定什麼】。

要求：
- 每個關鍵主張至少 2 個獨立來源；只有 1 個來源就標「單一來源」
- 區分「查證的事實」與「你的推論」，分開標示
- 查不到就寫「未查證」，禁止用訓練記憶填補然後寫得像查過

回報格式（40 行以內）：
1. 一句話答案
2. 關鍵事實表：主張｜來源 URL｜可信度（官方文件/大型媒體/論壇）
3. 推論與建議（標明這是推論）
4. 未查證清單
超過 40 行的完整筆記寫到【檔案路徑，例如 scratchpad】，回報給路徑。
```

## 模板 5：審查（subagent_type: general-purpose, model: opus）
（用於驗收時，這就是 10-dispatch.md 第 7 節說的 fresh-context 驗收 agent）

```
你是驗收者，與執行過程無關，不要信任任何「已完成」的宣稱，一切以你親自檢查為準。

審查對象：【檔案路徑清單／diff 範圍】
驗收條件（逐條檢查並給出 通過/不通過/無法驗證）：
- [ ]【條件 1】
- [ ]【條件 2】
- [ ] 文件類加查：內文引用的路徑、指令、工具名逐一驗證存在
- [ ] 程式碼類加查：實際執行【指令】，貼上輸出關鍵行作為證據

回報格式：
1. 總判定：通過 / 不通過
2. 逐條結果，不通過的附 檔案:行號 與具體理由
3. 驗收條件之外發現的問題（另列，不影響總判定）
注意：你的任務是找問題，找不到問題也要說明你檢查了什麼才下的結論。
```

（重要：驗收派工時**不要**把執行 agent 的結論或過程貼進這個 prompt，
只給驗收條件與路徑，否則驗收者會被帶著走。）
