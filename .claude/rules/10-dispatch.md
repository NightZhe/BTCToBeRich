# 10 - 模型調度守則

適用對象：主對話模型（任何等級）。目的：主對話的 context 是最貴的資源——它的每
一個 token 會被之後每一步重複攜帶。把大量讀寫推給 subagent，主對話只留結論。

---

## 0. 三問自檢（每次要動手做「非三分鐘小事」之前，先在心裡過一遍）
1. 這一步會拉多少內容進主對話？超過第 1 節門檻 → 派 subagent。
2. 派工 prompt 裡，三件套（目標與動機／驗收條件／回報格式）都寫了嗎？
3. 完成後誰來驗收？（不能是執行者自己——見第 7 節）

## 1. 指揮官不下場
主對話只做五件事：理解需求、拆解、派工、整合結論、對使用者溝通。

以下情況**一律派 subagent**，主對話不自己動手：
| 情況 | 門檻 | 派給 |
|------|------|------|
| 讀多個檔案 | 預計讀 ≥3 個檔案且合計 >100 行，或單檔 >500 行而你只需要其中一段 | Explore |
| 探索式搜尋 | 不知道目標在哪，要 grep/glob 掃 repo | Explore |
| 網頁搜尋/閱讀 | 任何需要看網頁全文的任務 | general-purpose |
| 批次改檔 | 要改 ≥3 個檔案 | general-purpose；多 agent 平行改檔必須用 `isolation: "worktree"` |
| 長輸出分析 | 跑測試/建置且輸出預期很長 | general-purpose，只回摘要與失敗清單 |

**例外（可以自己做）**：讀 1-2 個已知路徑的檔案、單檔小修改、跑單一指令看短輸出、
回答不需查證的問題。判斷標準：做完後留在主對話裡的內容 <100 行就自己做
（上表門檻與這條 <100 行標準衝突時，以 <100 行為準）。

**警告——滑坡**：「這次很小，直接做」連續三次，主對話就塞滿了。第二次想跳過
門檻時，就是該派工的訊號。

## 2. 可用資源（2026-07-07 於本環境查證；與 session 開頭系統注入不符時以注入為準，並按 40-maintenance.md 更新本節）
- **subagent 類型**（Agent 工具的 `subagent_type`）：
  `general-purpose`（萬用，可寫檔）、`Explore`（唯讀搜尋，回結論不回原文）、
  `Plan`（規劃架構）、`claude`（預設萬用）、`claude-code-guide`（Claude Code/API 使用問題）。
- **model 參數**（Agent 工具）：`haiku` | `sonnet` | `opus`。省略 = 繼承主對話模型。
  （參數表還有 `fable`，但不要把制度建立在它可用的前提上。）
- **effort 參數**：Agent 工具**沒有** effort 參數，不要寫了期待它生效。
  effort 只存在於 Workflow 的 agent() 呼叫（`low`~`max`），而 Workflow 需要使用者
  明說才能用（說「ultracode」或「用 workflow」）。沒說就用 Agent 工具。
- **完整型號 ID**（寫程式呼叫 API 時用）：`claude-sonnet-5`、`claude-opus-4-8`、
  `claude-haiku-4-5-20251001`。

## 3. 派工對照表（起手選擇；用第 6 節升降級修正）
| 任務型態 | subagent_type | model |
|----------|---------------|-------|
| 檔案定位、repo 掃描、「X 在哪裡」 | Explore | haiku |
| 機械式批次套用已知模式（改 import、換命名） | general-purpose | haiku |
| 一般實作、除錯、寫測試、網頁研究 | general-purpose | sonnet |
| 跨模組設計、兩次修不好的 bug、架構規劃 | Plan 或 general-purpose | opus |
| 驗收評審、高風險第二意見 | general-purpose | opus |

拿不準就 sonnet 起手。錯了按第 6 節調，不要事前糾結。

## 4. 派工三件套（每個派工 prompt 必含；完整模板見 30-templates.md）
1. **目標與動機**：要什麼＋為什麼要。動機讓 agent 在你沒料到的細節上做出對的取捨。
2. **驗收條件**：可逐條打勾的完成判準。寫不出驗收條件 = 你自己還沒想清楚，先想。
3. **回報格式**：明確規定回什麼、多長、什麼格式。

另外必附：**相關檔案的已知路徑**（省 agent 重新探索的 token）、**範圍邊界**
（哪些不要動）。多 agent fan-out 時，每個 agent 的範圍必須互斥，防止重工
（教訓：singapp 曾有兩個 agent 重複產出同一首歌的 JSON，見 lessons.md）。

## 5. 回報合約（寫進每個派工 prompt 的結尾）
- 只回：結論、關鍵證據（`檔案:行號`）、未解事項。
- 超過 30 行的產物寫到檔案裡，回報只給路徑＋3 行摘要。
- 禁止把大段程式碼或網頁原文貼回主對話。
- 主對話收到回報後：不重讀 agent 已讀過的檔案（驗收除外），不複述回報全文，
  只把「會影響下一步決策的部分」寫進給使用者的訊息。

## 6. 升降級路徑
- **haiku 失敗 1 次**（答非所問、改錯地方）→ 直接升 sonnet 重派。不要再給 haiku 第二次。
- **sonnet 同一子任務連錯 2 次** → 升 opus，且 prompt **必須**附完整失敗軌跡：
  試過哪些方法、確切錯誤訊息（原文貼）、當時的假設、為何判定失敗。
  只丟原始任務不附軌跡 = opus 會重走一遍相同的死路。
- **opus 解出來之後**，若同型任務還有多件 → 把解法寫成明確步驟清單，降回
  haiku/sonnet 批次套用。貴模型產出模式，便宜模型執行模式。
- **同一目標最多 2 輪重試**（換模型算新的一輪）。用完就停手，向使用者報告：
  卡在哪、試過什麼、剩下哪些選項。持續空轉比承認卡住更傷。

## 7. 驗證不自驗
執行者說「完成」只是宣稱，不是事實。收工前：
- **驗收一律派 fresh-context subagent**——一個沒有參與執行的新 agent。
  驗收 prompt 只給：驗收條件清單＋檔案路徑。**不給執行者的結論或過程**，
  否則驗收 agent 會照抄結論，驗證變形式。
- **檔案類**：驗收 agent read-back——檔案存在、逐條核對驗收條件、檢查內文引用的
  路徑/指令/名稱真實存在。
- **程式碼類**：跑測試或實際執行。沒有現成測試就寫最小重現腳本跑一次。
  「應該可以」「邏輯上正確」不算通過。
- **高風險或不可逆**（部署、刪檔、對外發送）：先過 20-judgment.md 判準 3
  （多半要問使用者），需要品質把關時加 opus 第二意見，或生成 2-3 個候選讓
  評審 agent 選優。
- **fan-out 收斂**：多 agent 平行工作後，主對話必須執行收斂步驟——合併結果、
  去重、清理 worktree。worktree 還掛著就不算完成。
